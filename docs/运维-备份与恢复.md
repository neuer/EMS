# 运维手册 — 数据库备份与恢复

适用：动环监控预警平台（PostgreSQL 15 + TimescaleDB，Docker Compose 内网单机部署）。

> 平台为纯只读对接 EMS，所有业务状态（资产镜像、元数据增强、规则、告警、通知配置、用户、报表计划等）均落在 PostgreSQL。Redis 仅缓存最新值/状态，可由推送与同步重建，**无需备份**。因此备份恢复以 PostgreSQL 为唯一对象。

## 1. 备份策略

| 项目 | 说明 |
|---|---|
| 工具 | `pg_dump -Fc`（自定义格式，支持选择性恢复与并行） |
| 内容 | 整库（含 TimescaleDB hypertable、连续聚合定义、元数据与配置表） |
| 频率建议 | 每日 1 次（低峰期，如 01:00），保留 14 天 |
| 存储 | 默认 `./backups/`，生产建议挂载到独立磁盘或同步到异机 |
| 校验 | 每个备份随附 `.sha256` 校验文件 |

### 手动备份
```bash
make backup
# 或自定义目录与保留天数
BACKUP_DIR=/data/backup RETENTION_DAYS=30 bash scripts/backup.sh
```
产物：`backups/dcim_YYYYMMDD_HHMMSS.dump` 与同名 `.sha256`。

### 定时备份（cron 示例）
在宿主机 `crontab -e` 加入（每天 01:10）：
```cron
10 1 * * * cd /opt/dh-monitor && BACKUP_DIR=/data/backup RETENTION_DAYS=30 bash scripts/backup.sh >> /var/log/dcim-backup.log 2>&1
```

## 2. 恢复流程（演练 / 灾备）

> ⚠️ 恢复为高风险操作：脚本会**删除并重建目标库**。务必确认备份文件正确、已通知相关人员。
>
> TimescaleDB 逻辑恢复必须走官方流程，否则 hypertable / 连续聚合会损坏。`scripts/restore.sh` 已内置：
> 重建空库 → `CREATE EXTENSION timescaledb` → `timescaledb_pre_restore()` → `pg_restore` → `timescaledb_post_restore()`。

```bash
# 1) 选定备份文件并校验（脚本会自动校验 .sha256）
ls -lh backups/

# 2) 执行恢复（默认交互确认；会自动停 backend → 恢复 → 起 backend）
make restore FILE=backups/dcim_20260620_010000.dump
# 自动化演练可跳过确认：
FORCE=1 bash scripts/restore.sh backups/dcim_20260620_010000.dump
```

恢复后核对：
1. 登录平台，确认用户/角色、规则、通知配置等数据完整。
2. 进入「系统设置 → EMS 连接」查看连接状态；必要时等待心跳重连或重启 backend。
3. TimescaleDB 连续聚合与保留策略随 dump 一并恢复；如趋势查询异常，确认 `point_history_5min` 物化视图存在。

## 3. 恢复演练自测清单（Sprint 7 验收项）

- [ ] `make backup` 生成 dump 且 `.sha256` 校验通过。
- [ ] 在测试库执行 `FORCE=1 scripts/restore.sh <dump>` 成功无报错。
- [ ] 恢复后登录正常、资产树/规则/告警历史可见。
- [ ] EMS 连接可重新在线并继续接收推送（对 `ems_mock` 验证）。

## 4. 回滚与应急

- 配置类误操作（规则/通知/用户）：可从最近备份单表恢复——
  `docker compose exec -T db pg_restore -U dcim -d dcim --clean --if-exists -t <表名> < <dump>`。
- 升级失败回滚：先 `docker compose down`（保留卷 `pgdata`），回退镜像/代码后再 `up`；数据卷不受代码回退影响。
- 数据卷损坏：以最近一次 dump 在新卷上 `restore`。

## 5. 注意事项

- 备份文件可能包含加密后的 EMS 凭据（Fernet 密文）与用户密码哈希（bcrypt），**不含明文**；仍应按敏感数据管理，限制访问权限。
- `ENCRYPTION_KEY` 必须与备份时一致，否则恢复后无法解密 EMS 凭据；该密钥需独立安全保管，切勿与备份同处存放。
