# 动环监控预警平台 — PRD 主文档

文档版本：v1.0 ｜ 面向：AI 编码 Agent（Claude Code）

---

## 1. 项目背景与目标

### 1.1 背景
机房现有动环监控由共济 EMS 承担数据采集。共济自带的上层平台臃肿、灵活性差、开放性不足，无法满足自定义预警、自定义展示与数据二次利用的需求。需自建一套轻量的上层集成平台，通过共济 EMS 的北向接口把全量动环数据「抓出来」，由我方自主掌控预警逻辑与呈现方式。

### 1.2 目标
- 通过共济北向协议完整接入单数据中心的空间/设备/测点配置、实时数据、历史数据与告警。
- 以**平台自有规则引擎**为核心，独立产生阈值/逻辑类预警，不依赖共济内置规则。
- 提供实时大盘、测点详情与趋势、空间拓扑、告警中心、NOC 大屏、报表与移动端访问。
- 全渠道告警通知，按告警级别路由。
- 轻量可控：单机 Docker Compose 内网部署，纯只读对接 EMS。

### 1.3 非目标（明确不做）
工单/派单、巡检、值班排班、资产台账、容量管理、能效 PUE、多数据中心、多厂商对接、对 EMS 的任何写/下控、向 EMS 反向下发告警规则、审计日志、等保合规基线建设。

## 2. 范围与边界

### 2.1 系统定位
平台等价于共济协议中的「上层动环数据接收平台 / 上层集成平台」，位于共济 EMS 之上。二者交互完全遵循《共济基础设施监控数据接入标准 V2.23》。

```
        [本平台：动环监控预警平台]
                  ▲  ▲
   (客户端：登录/心跳/拉取/历史) │  │ (服务端：接收实时数据/告警推送)
                  ▼  │
              [共济 EMS 北向接口]
                  ▲
        机房动力环境监控系统（设备/测点）
```

### 2.2 双角色说明（关键）
| 角色 | 平台行为 | 涉及接口 |
|---|---|---|
| HTTP **客户端** | 主动调用 EMS | login、heart、logout、get_space_list、get_device_list、get_spot_list、online_data（拉取）、offline_value（历史数据）、offline_alarm（历史告警）、get_event_rules（读规则，仅参考） |
| HTTP **服务端** | 接收 EMS 推送 | online_data_push（实时数据）、online_alarm_push（告警） |

> 推送独占：EMS 同一时刻仅向一个上层平台推送。本平台为唯一消费端，共济自带平台不再接收推送。

## 3. 名词术语
| 术语 | 含义 |
|---|---|
| 动环 | 动力环境（温湿度、配电、UPS、空调、漏水、消防等机房基础设施） |
| EMS | 共济动环监控系统（本平台的南向数据源） |
| 空间 / 设备 / 测点 | 共济对象模型三层：空间(ci_type=5)、设备(2)、测点(3)；测点是设备下的采集点 |
| 测点(Point) | 测点/控制点/告警点统称；本平台只读，关注测点实时值 |
| 实时值 | 测点当前采集值（real_value）+ 采集时间（save_time，Unix 秒） |
| 周期上传 | EMS 默认每 10s 全量推送一次测点数据 |
| 告警生命周期 | 产生→受理→确认→恢复（对应 msg_type 0/2/3/1） |
| 死区 | 数值变化未超过设定阈值则视为无变化，可选用于落库节流 |

| 角色 | 权限 |
|---|---|
| 管理员 admin | 全部功能 + 系统设置（EMS 连接、规则、通知渠道、用户管理） |

## 5. 功能需求（FR）

### FR-1 EMS 接入与连接管理
- **FR-1.1 登录鉴权**：以配置的 username/password 调用 `POST /north/login`，请求体携带本平台**接收推送的 ip/port**；成功后保存返回的 token，后续所有客户端请求在 Header 携带 `token`。
- **FR-1.2 心跳保活**：登录后启动心跳任务，**每 20s** 调用 `POST /north/heart`；EMS 30s 内收不到心跳即断连、token 失效。
- **FR-1.3 断连重连**：心跳失败 / token 失效（error_code=2 或 106）/ 推送长时间无数据时，自动重新登录获取新 token，并重新发起数据与告警订阅。重连采用指数退避（上限可配，如 60s）。
- **FR-1.4 单点登录**：同一用户多点登录会被 EMS 顶下线，平台保证全局单实例持有连接。
- **FR-1.5 连接状态可视**：在系统设置/状态页展示 EMS 连接状态（在线/离线）、最近心跳时间、最近推送时间、token 有效性、累计重连次数。
- **FR-1.6 优雅退出**：平台停止时调用 `POST /north/logout` 释放连接。

### FR-2 配置同步（空间 / 设备 / 测点树）
- **FR-2.1 全量同步**：登录后依次调用 `get_space_list`、`get_device_list`，并对设备分批调用 `get_spot_list`（resource_ids 数组），构建「空间→设备→测点」树并落库。
- **FR-2.2 同步时机**：平台启动时同步一次；之后**定时同步**（默认每 6h，可配）；并提供**手动「立即同步」**按钮。
- **FR-2.3 增量比对**：每次同步与本地对比，标记新增/变更/失效（消失的对象置 inactive，不物理删除，保留历史关联）。
- **FR-2.4 测点规则解析**：解析测点 `event_rules` 与 `filter` 字符串（分号键值格式），作为平台规则引擎初始化参考（不强制采用）。
- **FR-2.5 元数据本地增强**（不污染源数据）：对每个测点/设备/空间，平台侧可叠加：别名、自定义分组、标签（多个）、重要度（1–5）、自定义单位、备注。增强数据独立存储，与同步数据通过 resource_id 关联。

### FR-3 实时数据采集
- **FR-3.1 订阅推送**：登录后调用 `POST /north/online_data_subscribe`（subscribe=true）启动周期推送。
- **FR-3.2 接收推送**：实现服务端 `POST /north/online_data_push`，解析 `period / page_no / page_size / devices[] / points[]`，返回 `{"data":{"status":true}}`。分页包按 period 聚合为一轮。
- **FR-3.3 设备通信状态**：推送中 `device.status`（0 通信中断 / 1 正常）需落库并作为 EMS 设备级告警来源之一（见 FR-5）。
- **FR-3.4 最新值缓存**：每个测点最新 real_value/save_time 写入 Redis（供大盘/详情/WebSocket 低延迟读取）。
- **FR-3.5 历史落库**：实时值写入时序表（见 FR-4）。
- **FR-3.6 实时推前端**：通过 WebSocket 将订阅范围内测点的最新值实时推送给打开监控页的前端。
- **FR-3.7 空值兼容**：设备下无测点时 points 为 null，需正常处理不报错。
- **FR-3.8 在线拉取（兜底）**：实现客户端 `POST /north/online_data`（device_ids / spot_ids 二选一），用于手动刷新或推送异常时主动拉取。

### FR-4 历史数据与存储分层
- **FR-4.1 原始层**：实时数据按原始采集精度（约 10s）写入 TimescaleDB hypertable，**保留 30 天**（保留策略自动清理）。
- **FR-4.2 降采样层**：通过 TimescaleDB 连续聚合（continuous aggregate）生成 **5 分钟** 粒度聚合（avg/min/max），**保留 6 个月**。
- **FR-4.3 压缩**：原始层与降采样层启用列压缩，降低半年存储占用。
- **FR-4.4 趋势查询**：按时间范围与测点集查询；范围短取原始层、范围长自动取降采样层（查询路由）。支持多测点叠加。
- **FR-4.5 死区存储（可选，默认关）**：开启后，对模拟量仅在值变化超过测点级死区阈值时落原始层，并保留周期性心跳点防止曲线断裂；状态量逢变化即落。
- **FR-4.6 断连数据回补（必做）**：检测到推送中断（一段时间未收到 push）并恢复后，自动用 `POST /north/offline_value` 回补缺口。**严格遵守接口限制**：单次 ≤100 个测点、时间跨度 ≤1 天、同一时间仅一个历史请求在跑；故回补需按测点分批 + 按时间分片 + 串行执行，interval 取 five（5 分钟）写入降采样层，缺口标记已回补。

### FR-5 预警引擎（平台核心）
- **FR-5.1 混合告警源**
  - 平台规则引擎负责：阈值类（过高/过低/区间外）与逻辑类告警，基于测点实时值计算。
  - EMS 告警接入仅保留**设备级事件**：`event_type` 为 0（通信中断）、21（故障）、30（停止采集）；这些无法从测点值推导，订阅 `online_alarm_push` 后接收。
  - **去重**：对 EMS 推来的过高(2)/过低(4)/不正常值(3)等阈值类告警，默认丢弃（由平台引擎统一负责），避免与平台告警重复。去重策略可配。
- **FR-5.2 订阅 EMS 告警**：调用 `POST /north/online_alarm_subscribe`（subscribe=true）；实现服务端 `POST /north/online_alarm_push` 接收并解析告警结构（含 guid/msg_type/event_location/event_source/masked/event_alarm/event_recover/event_accept/event_confirm），返回 `{"data":{"status":true}}`。
- **FR-5.3 规则引擎（基础档）**
  - 规则对象绑定到测点；支持多条规则。
  - 运算符：`> < = <> <= >=`；条件类型：单阈值、区间（min/max）。
  - **多级阈值**：同一测点可配多档（如「次要>30、严重>35、紧急>40」），命中最高满足档。
  - **持续超限去抖**：连续满足条件达到 `continuous_time`（秒）才正式产生告警，避免抖动误报。
  - **自动恢复**：值回到恢复条件（restore_operator/restore_operand）并持续一定时间后自动置为恢复。
  - **5 级分级**：level 1–5，语义对齐附录（1 紧急、2 严重、3 重要、4 次要、5 提示）。
- **FR-5.4 告警生命周期**
  - 状态机：产生(active) → 受理(accepted) → 确认(confirmed) → 恢复(recovered)；亦支持「产生→恢复」直接闭环。
  - **人工受理/确认/备注**：操作员可受理、确认并填写处理备注，记录操作人与时间。
  - 同一逻辑告警的「产生/恢复/受理/确认」归并到一条告警记录的生命周期事件流。
- **FR-5.5 抑制能力（三项全要）**
  - **测点级屏蔽**：对指定测点屏蔽告警（屏蔽期内不产生平台告警，记录屏蔽态）。
  - **维护窗口静默**：可设维护时间窗（一次性或按计划）+ 适用范围（测点/设备/空间），窗口内命中规则不触发通知（可选择是否仍记录为「静默告警」）。
  - **防轰炸合并**：同一测点/同一规则在合并窗口内重复告警时合并计数，仅首次通知 + 周期摘要，避免短信轰炸。
- **FR-5.6 规则配置入口**：规则在平台侧 UI 配置与存储，由平台引擎计算。**不**通过 `set_event_rules` 反向写入 EMS。`get_event_rules` 仅用于「从 EMS 导入一份规则作为初值」的可选辅助。

### FR-6 通知
- **FR-6.1 渠道（全部支持，插件式）**：短信、邮件（SMTP）、钉钉（机器人 webhook）、企业微信（应用/机器人）、电话语音、通用 Webhook（自定义 HTTP 回调）。
  - 短信渠道对接**内部短信发送平台**（通过其 HTTP API 适配，密钥/网关地址可配）。
- **FR-6.2 按级别路由**：按告警 level 配置渠道映射，例：1–2 级 → 电话 + 短信；3 级 → 短信 + 钉钉；4–5 级 → 平台内 + 邮件。映射可在 UI 配置。
- **FR-6.3 接收人/接收组**：维护接收人（姓名 + 各渠道地址：手机号/邮箱/钉钉/企微 ID）与接收组；路由规则指向组或人。
- **FR-6.4 通知触发点**：告警产生、告警恢复（恢复通知可按级别开关）。
- **FR-6.5 防轰炸联动**：遵循 FR-5.5 合并窗口；通知失败重试（次数可配）并记录发送结果。
- **FR-6.6 不做**：未受理自动升级（escalation）暂不实现。

### FR-7 展示（全套视图）
- **FR-7.1 实时监控大盘**：KPI 卡片（在线设备数、活动告警数按级别、关键测点概览）+ 关键测点实时值面板（WebSocket 实时刷新）+ 重点区域状态。
- **FR-7.2 设备/测点列表 + 测点详情**：可按空间/设备/分组/标签筛选与搜索；测点详情含当前值、单位、所属设备/空间、当前规则、实时曲线 + 历史趋势钻取。
- **FR-7.3 历史趋势查询**：选择测点集（支持多测点叠加）+ 时间范围（快捷：近 1h/6h/24h/7d/30d/自定义）；自动按范围选原始/降采样层；ECharts 折线；支持导出 CSV。
- **FR-7.4 空间拓扑导航**：按空间树（区域→园区→楼宇→楼层→房间→机柜列→机柜位）逐层下钻，节点上叠加该范围内告警状态与关键指标。
- **FR-7.5 告警中心**：活动告警列表 + 历史告警查询；筛选（级别/类型/设备/空间/状态/时间/是否屏蔽）；受理、确认、备注操作；点击查看生命周期事件流。
- **FR-7.6 NOC 值班大屏**：适配机房值班墙的全屏大屏视图（深色主题、大字号、关键指标 + 实时告警滚动 + 告警级别统计 + 关键测点曲线），支持轮播与自适应分辨率。
- **FR-7.7 移动端/响应式**：核心页面（大盘、告警中心、测点详情）响应式适配手机浏览器，便于值班/外出查看与处理告警。

### FR-8 报表与导出
- **FR-8.1 告警统计报表**：按级别/设备/空间维度统计告警数量、恢复时长、Top 告警测点；支持日报/周报/月报周期。
- **FR-8.2 数据导出**：测点历史数据导出 CSV/Excel（指定测点集 + 时间范围 + 粒度）；告警记录导出。
- **FR-8.3 定时邮件报表**：可配置定时任务（如每日 08:00 发日报、每月 1 日发月报）自动生成并邮件推送给指定接收组。

### FR-9 系统设置（管理员）
- EMS 连接配置（地址/端口/账号/密码/本平台接收 ip:port/同步周期/订阅开关）。
- 规则管理（增删改查测点规则、多档阈值、去抖/恢复参数）。
- 通知配置（渠道参数、级别路由、接收人/组）。
- 元数据增强管理（别名/分组/标签/重要度/单位）。
- 维护窗口与屏蔽管理。
- 用户与角色管理。
- 报表计划管理。
- 存储策略开关（死区存储开/关及阈值）。

## 7. 关键约束（来自共济协议，务必遵守）
1. 报文统一格式：请求 `{"version":"...","data":{...}}`；响应 `{"error_code":0,"error_msg":"ok","data":{...}}`；UTF-8；值为空时键必须存在。
2. 鉴权：除 login 外所有请求 Header 携带 `token`。
3. 心跳：建议 20s 一次，30s 超时即失效需重登。
4. 历史接口限制：`offline_value` 单次 ≤100 测点、跨度 ≤1 天、同一时间仅响应一个历史请求。
5. 推送方向：实时数据/告警由 EMS POST 到平台登录时声明的 ip:port，平台必须实现对应服务端并返回 `status:true`。
6. 对象 ID：resource_id 全局唯一；空间/设备/测点 ci_type 分别为 5/2/3；测点 spot_type 1 模拟量 / 2 状态量 / 3 控制量。
7. 告警语义：event_type、event_level、msg_type 取值见《标准 V2.23》附录与 `03_API规格.md`。

## 8. 验收标准（关键）
- 平台启动后能登录共济（mock）、维持 20s 心跳、断连可自动重连，状态页正确反映连接信息。
